# 1 UART 串口通信
## 1.1 通信流程

![[Pasted image 20260703112029.png|384]]
- 起始位：发送方发送一个逻辑“0”信号，表示传输字符的开始。
- 数据位：数据位可以是5、6、7、8、9位等组成一个字符（通常是8位）
- 奇偶校验位：“1”位的个数应为偶数或奇数，以验证数据传输的正确性。
- 停止位：它是字符数据的结束标记。它可以是 1 位、1.5 位或 2 位。
- 常用：8N1。
## 1.2 过采样

异步通信如果只按波特率频率采样一次,容易采到信号跳变沿附近的不稳定区域。
解决方案:用**比波特率高N倍的时钟**持续采样,找到稳定的"中间点"。

STM32 默认为16倍过采样。
`USART_CR1` 寄存器的 `OVER8` 位:

| 模式           | 采样点      | 特点           |
| ------------ | -------- | ------------ |
| 16倍(OVER8=0) | 第8/9/10次 | 精度高,默认模式     |
| 8倍(OVER8=1)  | 第4/5/6次  | 支持更高波特率,抗噪略弱 |

## 1.3 UART 寄存器

### 1.3.1 DR / TDR / RDR  发送接收寄存器
  
在部分 STM32 系列中，USART_DR 表面上是一个寄存器地址，但读写方向不同：写 DR 等效写入发送数据寄存器，读 DR 等效读取接收数据寄存器。新系列 STM32 通常显式拆分为 TDR 和 RDR。  
TDR/RDR 只是"缓冲区",真正一位一位往外/往里移位的是硬件里的 **TSR/RSR**(Transmit/Receive Shift Register),这两个移位寄存器程序访问不到。
### 1.3.2 TXE  发送寄存器数据为空
  
TXE 表示发送数据寄存器为空（缓冲区为空），即 TDR 已经可以写入下一个数据。代表发送缓冲区已经空出来。此时上一个字节可能仍在移位寄存器中继续发送。  

连续发送多个字节时，通常依靠 TXE 判断是否可以继续写入下一个字节，从而提高发送效率，避免字节之间出现空隙。

应用途径：
- 阻塞/DMA/中断发送数据

### 1.3.3 TC  发送寄存器数据为空(Transmission Complete)

TDR为空 **并且** TSR也为空——也就是说,包括停止位在内的整个帧已经**真正从物理线上发出去了**，此时是真正发送完成。

如果你只等TXE就认为发送"完成"了去做后续动作,这时候TSR里其实还有数据正在往外移位(尤其是停止位还没发完),线路还没真正空闲。

应用途径：
- RS485发送完把方向控制引脚(通常接在收发器的DE/RE)切回接收模式。**这个切换动作必须等TC=1才能做。
### 1.3.4 RXNE
`RXNE`(Receive Data Register Not Empty)—— 接收数据寄存器非空**,表示RDR里已经有一个完整字节等着你去读了。和TXE是镜像关系

| |TXE|RXNE|
|---|---|---|
|关心的寄存器|TDR(发送缓冲)|RDR(接收缓冲)|
|置1条件|TDR被搬空了(可以写下一个)|RDR被填满了(有数据可读)|
|清除方式|写DR自动清|读DR自动清|
|对应移位寄存器|TSR(往外发)|RSR(往里收)|
## 1.4 IDLE
**IDLE判定的空闲时长 = 1个字符时间(约等于一帧的传输时间)**,不是固定的绝对毫秒数,而是随波特率和帧格式变化的。空闲中断的触发条件是从收到上一个字节完成起(RXNE=1)算起,超过一个字节周期的时长没有收到新数据,才会产生硬件中断。

# 2 SPI 通信

## 2.1 硬件层

SPI 的四条线分别是：

- **MOSI**（Master Output Slave Input）：主发从收
- **MISO**（Master Input Slave Output）：主收从发
- **SCLK**（Serial Clock）：时钟信号线
- **CS**（Chip Select，或称 SS，Slave Select）：片选

主从架构，**没有地址机制**——靠 CS 线的物理连接来区分设备，这也是 SPI 和 I2C 最大的设计差异之一。

> 全双工：MOSI 和 MISO 两根线各走一个方向，可以同时收发。这是 SPI 和半双工的 I2C 的另一个关键区别。

## 2.2 通信流程

**CPOL（Clock Polarity）**：决定 SCLK 空闲电平

- CPOL=0：空闲低电平
- CPOL=1：空闲高电平

**CPHA（Clock Phase）**：决定第几个时钟边沿采样

- CPHA=0：第一个边沿采样
- CPHA=1：第二个边沿采样

|模式|CPOL|CPHA|空闲电平|采样边沿|
|---|---|---|---|---|
|Mode 0|0|0|低|上升沿|
|Mode 1|0|1|低|下降沿|
|Mode 2|1|0|高|下降沿|
|Mode 3|1|1|高|上升沿|

最常用：**Mode 0 和 Mode 3**。主从模式不一致 → 采样时机对不上 → 通信要么完全不通，要么收到数据整体错位/乱码，是调 SPI 最常见故障之一。

![[Pasted image 20260703131146.png]]

用 Mode 0 举例完整走一遍：

```
CS拉低
       ↓
数据提前放到MOSI/MISO上(准备阶段)
       ↓
SCLK第一个上升沿 → 采样(读取这个已经稳定的值)
       ↓
SCLK下降沿 → 改变数据,准备下一位
       ↓
SCLK第二个上升沿 → 采样下一位
       ↓
...以此往复
```

**采样边沿和改变边沿是交替进行的**。

## 2.3 移位寄存器：数据到底怎么传的

**本质：主从双方各有一个移位寄存器，通信时两个寄存器互相倒数据。**

```
MOSI: 主机 ──→ 从机
MISO: 从机 ──→ 主机
同一个 SCLK 边沿,两根线各移出/移入 1 位
8 拍之后,两个寄存器的内容完全交换
```

**一次交换的完整流程（读、写都是这个骨架）：**

```
① 装填:双方各自往 DR(数据寄存器)写这一轮要发的数据
        主机写命令或 dummy,从机写它要回复的值
② 移位:8 个 SCLK,MOSI/MISO 同时逐位移出、移入
③ 卸载:双方读走各自收到的数据
```

**三个反直觉但必须记牢的点：**

- 移位寄存器**不是靠「读」变空的**，是靠下一轮「写」重新装满。顺序永远是 **读走收到的 → 写入要发的 → 移位**。
- **硬件不会自己循环**。打完 8 拍就停，第 2 轮寄存器里装的是软件新写进去的值，绝不是上一轮的残留。所以「再传一轮会不会把数据又转回去」是伪问题。
- 从机能把数据发回主机，靠的是**从机提前把要回的值压进自己的 DR**（装填），不是什么反向传输机制。全双工物理上就是两根线各走一个方向而已。

# 3 BMI088 通信

## 3.1 基本结构

**Accel 和 Gyro 是两个独立的部分，各有独立的片选（CS）和独立的寄存器组**，必须分开访问，不能一次 burst 把 6 轴全读出来。

- 上电默认接口：Gyro 直接可用 SPI；**Accel 默认是 I2C**，SPI 首次读前需要做一次 dummy read 把它切换到 SPI 总线（初始化时做一次即可，与下面的读数 dummy 不是一回事）。

## 3.2 读时序差异（核心考点）

```
Gyro 读: [命令|读位][dummy]        → 取第 2 字节   无 dummy 延迟
Accel 读:[命令|读位][dummy][data]  → 取第 3 字节   ★ 多一个 dummy 字节
```

**为什么 Accel 要多一拍：** 它内部从「收到地址」到「把数据准备好」需要时间，来不及在紧接着的字节里回复，于是第 2 字节先吐一个 dummy 占位，第 3 字节才是真实数据。

**「帧间」根本不用管：** CS 全程拉低，一口气发完所有字节。dummy 是主机自己塞进发送缓冲、又自己从接收缓冲丢掉的。代码上就是**数组多开一格、取数索引往后挪一位**。


## 3.3 数据位宽与字节数

```
陀螺仪、加速度计都是 16 位有符号 (int16_t)
但寄存器是 8 位 → 每个轴拆成 2 字节: [高字节 MSB][低字节 LSB]
```

- **MSB**（高字节）= 16 位数左边、权重大的 8 位；**LSB**（低字节）= 右边、权重小的 8 位。
- **BMI088 是小端**：低字节（LSB）地址在前、先被读出。

**字节数计算（注意单位）：**

```
16 位 = 2 字节  (不是 16 字节!)
三轴角速度 + 三轴加速度 = 6 轴 × 2 字节 = 12 字节
```


# 3 I2C

## 3.1 SCL和SDA
I2C是**同步、半双工、多主多从**总线协议,只用2根线就能挂载多个设备。

| 信号  | 特点           |
| --- | ------------ |
| SDA | 双向数据线        |
| SCL | 时钟线,通常由主设备产生 |

I2C的SDA和SCL都是**开漏(Open-Drain)输出**,只保留了**下拉那一个晶体管(NMOS)**,输出低电平时NMOS导通、把线拉低;但**输出"高电平"时,NMOS直接截止、悬空(呈高阻态)**，这个特性直接支撑了I2C的三大机制：

1. 多设备共享2根线不冲突。

2. 时钟延展(Clock Stretching)**——从设备之所以能"抢过"SCL的控制权、强行把它拉低拖住主设备,靠的正是开漏输出的这个特性:从设备和主设备都能拉低SCL,谁在拉低,总线就是低电平,主设备就算想把SCL释放成高电平,只要从设备还在拉着,线路就是低——**这在推挽输出上是做不到的**,因为主设备会强行把线路拉高,和从设备的拉低直接冲突。

3. 多主仲裁——仲裁的本质就是利用"线与"特性:某个主设备想发1(即放手,不拉低),但如果同时另一个主设备在发0(拉低),线路电平就会被拉成0。发1的那个主设备读回总线电平发现和自己发的不一致(自己想发1,读到的却是0),就知道自己仲裁失败,乖乖退出。**这套机制完全建立在"没人能强行拉高线路"这个开漏输出的前提之上**——如果是推挽,两个主设备同时输出不同电平会直接短路,而不是"自然形成正确的仲裁结果"。

由于 I2C 的 SDA 和 SCL 都是开漏输出结构，设备只能主动把总线拉低，不能主动输出高电平。SDA 和 SCL 需要依靠外部上拉电阻被拉回高电平。

## 3.2 上拉电阻与总线电容

SCL和SDA上升时间与时间常数τ=RC有关，RC越大，上升时间越长，反之越短。
### 3.2.1 总线电容Cb

总线电容 = 所有挂载设备的引脚电容之和 + PCB走线寄生电容。设备越多、走线越长,电容越大,充电越慢。所以IIC没法做到长距离通信，主要是因为Cb。
在电路设计上，尽可能减少Cb，Cb越小，可做到的距离和通信质量可以越高。
一般来说：
10 cm ~ 20 cm：基本稳  
30 cm：普通板间连接较常见  
50 cm：开始需要谨慎
### 3.2.2 上拉电阻

上拉电阻通常连接在 SDA/SCL 与电源 VCC 之间，其作用主要有三个：

1. 提供总线高电平  
   当总线上没有设备拉低 SDA 或 SCL 时，上拉电阻将总线拉到高电平，使总线处于空闲状态。

2. 限制灌电流  
   当某个设备把总线拉低时，电流从 VCC 经过上拉电阻流入该设备的 NMOS 管到地。上拉电阻可以限制这个电流，避免电流过大损坏器件。

3. 影响信号上升沿速度  
   I2C 总线从低电平变为高电平时，并不是由器件主动拉高，而是通过上拉电阻给总线寄生电容充电。因此，上拉电阻和总线电容共同决定了信号的上升时间。

但是阻值太小会带来两个新问题:
1. **下拉功耗增大**:设备拉低总线时相当于电源经上拉电阻直接对地放电,电阻越小瞬时电流越大,对电池供电设备是明显开销。
2. **拉低电平不够低**:开漏输出内部的NMOS导通电阻不是0,如果上拉电阻和它的比值不够悬殊,拉低时的电压是两者分压的结果,可能达不到规范要求的最大低电平门限,导致逻辑判断错误。

**结论**:上拉电阻选型是"上升沿够快"和"功耗、低电平门限不能牺牲"之间的权衡,不存在越小越好这种简单结论。

## 3.3 通信流程

![[Pasted image 20260703150214.png|532]]
I2C 总线空闲时，SDA 和 SCL 都为高电平。
可以这样理解：SCL为高电平有效，高电平的时候进行检测（检测上下边沿，检测数据高低）
### 3.3.1 起始信号与停止信号

> [!important]  
> 起始和停止都是由主机发送，在SCL为高电平的时候，SDA进行上下边沿变化

当 SCL 保持高电平时，==SDA 从高电平跳变为低电平==，表示起始信号 START。
START 信号由==主设备产生==，用于通知总线上的所有从设备：一次新的通信即将开始。

当 SCL 保持高电平时，==SDA 从低电平跳变为高电平==，表示停止信号 STOP。
STOP 信号由==主设备在通信完成后产生==，用于释放总线。STOP 之后，总线重新回到空闲状态，即 SDA 和 SCL 都为高电平。

在起始条件产生后，总线处于忙状态，其他I2C器件无法访问总线；

### 3.3.2 数据有效性

> [!important]  
> 与起始和停止不同的是，在SCL为高电平的时候，SDA不能变化，SDA的值就是传输的值，在SCL中间采样

数据的传输是在发送完成了启动信号之后便可以进行数据的传输了。

IIC协议要求在时钟信号SCL为高电位的期间，数据线SDA上的数据要保持稳定，不能发生变化。只有在时钟信号SCL电位变低的时候，数据线SDA上的电平状态才能发生跳变。  

发送到 SDA 线上的每个字节必须为 8 位，每次传输可以发送的字节数量不受限制 ，每个字节后必须跟一个响应位 ，首先传输的是数据的最高位（MSB ）。

如果从机要完成一些其他功能后（如中断）才能接收或发送下一个完整的数据字节 ，可以使时钟线 SCL 保持低电平迫使主机进入等待状态 ，当从机准备好接收下一个数据字节时释放时钟线 SCL， 数据传输继续。

### 3.3.3 响应ACK

数据传输必须带响应，相关的响应SCL时钟脉冲由主机产生，在响应的时钟脉冲期间，发送器释放 SDA 线（输出高阻态使SDA线被上拉电阻拉高）。在响应的时钟脉冲期间，接收器必须将 SDA 线拉低，使它在这个时钟脉冲的高电平期间保持稳定的低电平。

### 3.3.4 寻址
![[Pasted image 20260703160339.png|559]]

> [!important]  
> 地址帧前面一定要有START条件，如果不发START，后面字节不会被解释为地址帧，其他设备不会监听你的寻址信息

==在起始条件之后==，发送一个从机地址，这个地址共 7 位，紧接着的第 8 位是数据方向位R / W， 0 表示写，1表示读。

接下来的一个bit是应答位NACK/ACK，当这个帧中前面8bits发送完后，接收端获得SDA控制权，此时接收设备应该在第9个时钟脉冲之前回复一个ACK（将SDA拉低）以表示接收正常，如果接收设备没有将SDA拉低，则说明接收设备可能没有收到数据（如寻址的设备不存在或设备忙）或无法解析收到的消息，如果是这样，则由master来决定如何处理（stop或repeated start condition）。

## 3.4 仲裁

I2C 支持多主机结构。多个主设备都可以在总线空闲时尝试发起通信，但同一时刻总线上只能有一个主设备真正获得控制权。

I2C 总线空闲时，SDA 和 SCL 都为高电平。主设备在发起通信前，需要先检测总线是否空闲。

### 3.4.1 非同时发起 START

如果某个主设备已经先产生 START 条件，则 SDA 会在 SCL 高电平期间被拉低，总线进入忙状态。此时其他主设备检测到总线忙后，应等待当前通信结束，即等待 STOP 条件出现后，再重新尝试发起通信。

因此，如果不是多个主设备几乎同时发起 START，通常不会进入复杂的位级仲裁过程。可以简单理解为：谁先成功产生 START，谁先占用总线。

### 3.4.2 同时发起 START

如果两个或多个主设备几乎同时检测到总线空闲，并同时产生 START 条件，那么总线上仍然只会形成一个合法的 START 信号。此时多个主设备都可能认为自己已经获得总线控制权，随后会在地址位、读写位或数据位传输过程中进行仲裁。

I2C 的仲裁发生在 SCL 为高电平期间。主设备在发送 SDA 的同时，也会读取总线上的 SDA 实际电平。

由于 I2C 是开漏结构，发送 0 表示主动拉低 SDA，发送 1 表示释放 SDA。因此，当一个主设备发送 0，另一个主设备发送 1 时，总线实际电平会被拉低为 0。

如果某个主设备想发送 1，但读回 SDA 却是 0，说明有其他主设备正在发送 0，该主设备仲裁失败，需要退出竞争。==因此在仲裁过程中不会丢失信息==。

## 3.5 读取完整流程

### 3.5.1 主机寻址+声明写方向
```
第1步:主机发送START
第2步:主机发送 7bit从机地址 + R/W位(此处W=0,即先按"写"方向寻址)
第3步:从机ACK(表示"地址匹配,我在,准备好了")
```
### 3.5.2 声明要读哪个寄存器
```
第4步:主机发送寄存器地址(比如要读的是哪个寄存器编号)
第5步:从机ACK（从机把指针指向寄存器那了）
```
### 3.5.3 切换方向
```
第6步:主机再发一次START(重复起始,因为要寻址了)
第7步:主机再发一次 7bit从机地址 + R/W位(这次R/W=1,切换成"读"方向)
第8步:从机ACK（表示"收到，待会我来发"）
```
### 3.5.4 阶段三:实际读数据+结束
```
第9步:从机把数据放到SDA上,主机读取1字节(8bit)
第10步:主机产生ACK（告诉从机"收到了，我还要"）
第11步:读CRC校验码(部分带CRC校验的芯片才有这一步,比如Sensirion的传感器)
第12步:主机产生NACK（告诉从机"我不要了，够了"）
第13步:主机发送STOP,释放总线
```
## 3.6 故障排除：总线锁死与恢复

### 3.6.1 什么是总线锁死

I²C 总线锁死通常指 `SDA` 或 `SCL` 被某个器件持续拉低，导致主机无法重新发起通信。
最常见情况是：主机和从机通信过程中，主机突然复位或通信被异常打断。此时从机可能还停留在原来的通信状态机中，等待主机继续提供 `SCL` 时钟，因此一直拉低 `SDA`。

如果 `SDA` 已经被从机持续拉低，主机就无法产生有效的 `START` 条件，表现为 I²C 总线卡死。

### 3.6.2 怎么恢复

恢复思路是：先关闭 I²C 外设，把 `SCL` 和 `SDA` 临时切换成 GPIO，再手动产生时钟，让从机退出异常状态。

```text
1. 关闭 I²C 外设
2. 将 SCL、SDA 配置为 GPIO 开漏模式
3. 释放 SCL、SDA
4. 手动产生 9 个 SCL 时钟
5. 手动产生 STOP 条件
6. 重新初始化 I²C 外设
7. 重新发起通信
```

手动产生 9 个 `SCL` 的原因是：一个 I²C 字节由 `8 bit 数据 + 1 bit ACK/NACK` 组成。补足时钟后，从机有机会完成当前字节传输并释放 `SDA`。

如果恢复后 `SDA` 或 `SCL` 仍然被拉低，说明从机可能已经进入更严重的异常状态，此时需要复位从机，或者对从机断电重启。

实际工程中，I²C 驱动必须给等待标志位的地方加超时。发生超时后，不能一直死等，应退出当前通信并执行总线恢复流程。
