
# 1 C/C++ 与内存

## 1.1 指针与关键字

1. 指针数组与数组指针的定义及区别，分别如何使用？
	- **指针数组** `int *p[4]`：本质是**数组**，存 4 个 int 指针（`[]` 优先级高于 `*`）。用途：存多个字符串，如 `char *name[3]`。
	- **数组指针** `int (*p)[4]`：本质是**指针**，指向"含 4 个 int 的数组"。用途：指向二维数组的一行，`p++` 跳过一整行。
2. 嵌入式中避免野指针的方法。
	- 定义时初始化为 `NULL`；使用前判空。
	- `free`/`delete` 后立即置 `NULL`。
	- 不返回局部变量的地址；防止数组越界写坏指针。
	- 指针作用域尽量小，谁分配谁释放。
3. `const` 关键字的用法。
	- 修饰变量：只读，编译期保护，防止误改。
	- 修饰函数参数：`const char *p`，承诺函数内不改指向的内容。
	- 修饰返回值、修饰 C++ 成员函数（函数内不能改成员变量）。
	- 嵌入式意义：`const` 全局变量放 Flash（.rodata 段），**省 RAM**。
4. `const` 修饰指针时，“指针常量”与“常量指针”的本质差异是什么？
	- **常量指针**（指向常量的指针）`const int *p`：**值不能改**，指针可换指向。
	- **指针常量** `int *const p`：**指针不能换指向**，值可以改。
	- 记忆：const 在 `*` 左边锁"值"，在 `*` 右边锁"指针"。`const int *const p` 两者都锁。
5. `delete[]` 和 `delete` 的区别。
	- `delete`：释放单个对象，只调 1 次析构函数。
	- `delete[]`：释放数组，对每个元素依次调用析构函数。
	- 必须配对：`new` 配 `delete`，`new[]` 配 `delete[]`，混用是未定义行为（类类型会漏调析构、可能内存泄漏或崩溃）。

## 1.2 内存与数据布局

1. C 语言程序运行时的内存分配。
	- **代码段 .text**：程序指令，只读。
	- **只读数据段 .rodata**：常量、字符串字面量。
	- **数据段 .data**：已初始化的全局/静态变量。
	- **BSS 段**：未初始化（或初值为 0）的全局/静态变量，启动时清零。
	- **堆**：`malloc`/`free` 手动管理，向高地址增长。
	- **栈**：局部变量、函数参数、返回地址，向低地址增长。
2. 堆和栈有什么区别？堆区和栈区之间还有什么区？
	- 管理：栈由编译器自动分配释放；堆手动 `malloc`/`free`。
	- 大小：栈小（MCU 上常为几 KB）；堆相对大。
	- 速度：栈快（只动 SP 指针）；堆慢（查找空闲块）。
	- 碎片：堆有碎片问题，栈没有。增长方向相反（栈向下、堆向上）。
	- 中间还有：全局区（.data/.bss）、常量区（.rodata）、代码区；严格说堆和栈之间是一段未使用的空闲内存，两者相向增长，撞上即溢出。
3. 内存泄漏的原因。
	- `malloc`/`new` 后没有 `free`/`delete`。
	- 指针被覆盖，丢失了原来分配的地址。
	- 函数提前 return / 异常路径跳过了释放代码。
	- C++ 智能指针循环引用。
	- 排查：valgrind、封装 malloc 做计数统计、代码 review。
4. 未初始化的全局变量放在哪里？BSS 段省的是什么空间？
	- 放 **BSS 段**（RAM 中）。
	- 省的是 **Flash（可执行文件）空间**：BSS 不存实际数据，镜像里只记"起始地址+大小"，启动时统一清零；而 .data 段必须在 Flash 中存一份初值。
5. 结构体对齐：`u8 u16 u32` 和 `u32 u16 u8` 分别占用多少字节？
	- 规则：每个成员对齐到**自身大小的整数倍**地址；总大小补齐为**最大成员**的整数倍。
	- `u8 u16 u32`：0(u8) + 1(填充) + 2~3(u16) + 4~7(u32) = **8 字节**。
	- `u32 u16 u8`：0~3(u32) + 4~5(u16) + 6(u8) + 7(填充) = **8 字节**。
	- 两者都是 8；但成员顺序不同确实可能导致大小不同（如 `u8 u32 u16` = 12）。
6. 如果 MCU RAM 不够了，你会如何分析内存占用？全局变量、堆、栈、RTOS 任务栈分别怎么排查？
	- **全局变量**：看 map 文件，按 .data/.bss 大小排序，找出大数组、大缓冲。
	- **堆**：统计分配峰值；FreeRTOS 用 `xPortGetFreeHeapSize()` 和 `xPortGetMinimumEverFreeHeapSize()`。
	- **栈**：栈区预填魔数（如 0xA5），运行后看高水位；主栈看 MSP 最深位置。
	- **任务栈**：`uxTaskGetStackHighWaterMark()` 逐任务查剩余量，裁剪给多了的。
	- 优化：大缓冲复用/改动态、const 表放 Flash、缩小队列和日志缓冲。

# 2 编译、启动、架构与调试

## 2.1 编译与启动

1. 程序从写完到运行经历什么阶段？
	- **预处理**：展开宏和头文件、删注释 → `.i`。
	- **编译**：翻译成汇编 → `.s`。
	- **汇编**：生成机器码目标文件 → `.o`。
	- **链接**：合并目标文件和库、地址重定位 → 可执行文件（elf/bin/hex）。
	- 嵌入式还要：烧录到 Flash → 上电从复位向量开始执行。
2. STM32 的启动流程？
	- 上电复位，硬件从 0x08000000（Flash 起始，由 BOOT 引脚决定映射）取前两个字。
	- 第 1 个字 → 装入 **MSP**（栈顶指针）；第 2 个字 → **Reset_Handler** 地址，跳转执行。
	- Reset_Handler：把 **.data 从 Flash 复制到 RAM**、**清零 .bss**、调用 `SystemInit()` 配置时钟。
	- 最后调用 `__main`/`main()` 进入用户程序。
3. 单片机运行时，Flash 中的什么数据要搬到 RAM 中？
	- **.data 段**：已初始化全局/静态变量的初值（Flash 存一份，启动时搬到 RAM）。
	- .bss 不用搬，只需在 RAM 清零。
	- 另外可能搬：要求高速执行的 **RAM 函数**（RamFunc）、重映射到 RAM 的中断向量表。
4. 说说 U-Boot 启动流程。
	- **第一阶段（SPL，汇编为主）**：关看门狗和中断 → 设置 CPU 模式 → 初始化时钟和 DDR → 把第二阶段代码从 Flash 拷到 RAM → 跳转。
	- **第二阶段（C 为主）**：初始化串口、存储、网络等外设 → 加载环境变量 → 倒计时，进命令行或自动引导。
	- **引导内核**：把 kernel 和设备树加载到 RAM，通过 bootargs 传参，跳转内核入口。
5. 为什么 Bootloader 分两个阶段？第一阶段为什么必须足够小？
	- 上电时 **DDR 还没初始化**，代码只能在 SoC 片内 SRAM 里运行，容量只有几十 KB。
	- 所以第一阶段必须小到能装进片内 SRAM（且满足 BootROM 的加载限制），只做最必要的事：时钟 + DDR 初始化。
	- DDR 可用后，把功能完整的第二阶段搬进大内存再运行。
6. 如果系统启动时间从 800 ms 优化到 300 ms，你会从 Bootloader、时钟初始化、外设初始化、任务创建、日志输出哪些地方入手？
	- **先测量**：GPIO 翻转 + 示波器（或时间戳日志）分段统计各阶段耗时，找大头。
	- **Bootloader**：裁剪功能、去掉倒计时、提高 Flash 读取速度（QSPI/DMA/提高分频）。
	- **时钟**：减少不必要的 PLL 稳定等待、避免重复配置。
	- **外设**：延迟初始化（用到再 init）、把上电自检的长延时改成异步、并行化。
	- **任务**：启动期只创建关键任务，非关键任务延后创建。
	- **日志**：降低启动期日志量、提高波特率、改异步日志。

## 2.2 MCU 与 ARM

1. ARM 架构：Cortex-M3、ARMv7 等相关内容。
	- Cortex-M3 基于 **ARMv7-M** 架构，哈佛结构（取指与取数分开），3 级流水线，Thumb-2 指令集。
	- 无 MMU（可选 MPU），带 **NVIC** 中断控制器，支持位带操作。
	- 寄存器：R0-R12 通用，R13=SP（分 MSP/PSP 双栈），R14=LR，R15=PC，xPSR。
	- 两种工作模式：Handler（异常）/ Thread（线程）；分特权级和非特权级。
2. 简单说一下用过的 STM32：主频、Flash、RAM。
	- 参考（换成自己实际用的型号）：**F103**：Cortex-M3，72 MHz，Flash 最大 512 KB，RAM 64 KB。
	- **F407**：Cortex-M4（带 FPU），168 MHz，Flash 1 MB，RAM 192 KB。
	- 加分：说清自己项目里用了多少 Flash/RAM、外设资源（定时器/串口/DMA 数量）。
3. STM32 中断控制器是什么？
	- **NVIC**（嵌套向量中断控制器），Cortex-M 内核的一部分。
	- 支持中断嵌套；优先级分组：抢占优先级 + 子优先级（4 位共 16 级可配，数值越小优先级越高）。
	- 通过中断向量表找 ISR 入口，进入异常时硬件自动压栈 8 个寄存器。
	- 外部 GPIO 中断先经过 EXTI 控制器再到 NVIC。

## 2.3 调试与故障定位

1. 用过 GDB 吗？常用命令有哪些？
	- 断点：`b`（break）、`watch` 观察点、`info b` 查断点。
	- 执行：`r` 运行、`c` 继续、`n` 单步跳过、`s` 单步进入、`finish` 跑完当前函数。
	- 查看：`p` 打印变量、`x` 查看内存、`bt` 调用栈、`frame N` 切栈帧、`info registers` 看寄存器。
	- 嵌入式：配合 OpenOCD / J-Link GDB Server，`target remote :3333` 远程调试。
2. 对寄存器的了解，如何查看运行过程中的调用栈？（打断点调试）
	- 打断点后看 SP、LR、PC；`bt` 或 IDE 的 Call Stack 窗口看调用链。
	- 原理：每次函数调用都会把返回地址（LR）压栈，沿栈帧回溯即得调用栈。
	- HardFault 时：从被压栈的 PC/LR 找出错代码位置，结合 SCB->CFSR/HFSR 判断故障类型。
3. 不用调试器怎么判断问题？（打日志、接万用表或示波器；也可以在问题发生时把上下文保存到 Flash 中）
	- 串口分级日志，缩小问题范围。
	- LED / GPIO 翻转标记程序执行到哪，示波器测时序。
	- 万用表/示波器/逻辑分析仪看硬件信号。
	- 故障钩子里把寄存器、栈、关键变量写入 Flash，重启后回读分析。
4. 访问内存出错、Bus Error 这类问题如何排查？
	- 读 Fault 寄存器：**CFSR**（含 BFSR/MMFSR/UFSR）判类型，**BFAR/MMFAR** 拿到出错地址。
	- 从异常栈帧取出错时的 PC，对照反汇编/map 文件定位到代码行。
	- 常见原因：空指针/野指针、数组越界、**栈溢出**、访问未使能时钟的外设、未对齐访问、DMA 写坏内存。
	- 栈溢出验证：看 SP 是否越过栈底、任务栈高水位。
5. 系统运行几个小时后偶发卡死，但现场无法接调试器，你会在固件里提前设计哪些日志、异常记录和自恢复机制？
	- **看门狗**：独立看门狗兜底，喂狗与任务心跳挂钩（哪个任务死了能定位到）。
	- **异常现场保存**：HardFault/断言钩子里把寄存器、栈、当前任务名写 Flash 或后备 SRAM，重启后上报。
	- **常驻环形日志**：关键事件记在 RAM 环形缓冲，崩溃后可回读；记录复位原因寄存器（RCC->CSR）。
	- **运行监控**：任务栈高水位、堆余量、malloc 失败钩子、CPU 占用率定期记录。
	- **自恢复**：分级处理——模块复位 → 软件复位；重启后带着故障码继续上报。

# 3 外设、总线与通信协议

## 3.1 I²C

1. I²C 多主仲裁。
	- SDA 是开漏线与结构：**谁发 0 谁赢**（低电平优先）。
	- 每个主机边发边回读 SDA：自己发 1 却读到 0，说明仲裁失败，立即退出（转从机或等总线空闲重试）。
	- 仲裁在数据第一个不同位分出胜负，且**不破坏总线上的数据**（赢家的数据完整发出）。
	- 配套机制：时钟同步——SCL 低电平取各主机中最长者。
2. I²C 的通信时序。
	- **起始**：SCL 高电平时 SDA 下降沿；**停止**：SCL 高电平时 SDA 上升沿。
	- **数据有效性**：SCL 高电平期间 SDA 必须保持稳定，只能在 SCL 低电平时变化。
	- 每传 8 位后跟 1 个 **ACK 位**（接收方拉低 SDA 应答）。
	- 完整帧：Start → 7 位地址 + 读写位 → ACK → 数据字节 + ACK…… → Stop。
	- 速率：标准模式 100 kbps，快速模式 400 kbps。

## 3.2 SPI

1. 讲一讲 SPI 通信中的 CPOL（时钟极性）和 CPHA（时钟相位）。
	- **CPOL**：SCK 空闲时的电平，0=空闲低电平，1=空闲高电平。
	- **CPHA**：采样时刻，0=第 1 个时钟边沿采样，1=第 2 个边沿采样。
	- 组合成 4 种模式（Mode0~3），**主从双方必须一致**，否则数据错位。
	- 最常用 Mode0（CPOL=0, CPHA=0）和 Mode3；具体看从机芯片手册。
2. SPI 有几根线？通信过程是什么？
	- 4 根：**SCK** 时钟、**MOSI** 主出从入、**MISO** 主入从出、**CS** 片选（多从机时每个从机一根 CS）。
	- 过程：主机拉低 CS 选中从机 → 主机发时钟，主从的**移位寄存器同步交换数据**——主机每发出 1 字节同时收回 1 字节（全双工）→ 拉高 CS 结束。
	- 想只读时主机发 dummy 字节来产生时钟。
3. 如果 SPI 接口连接 IMU，采样频率很高，你会如何设计 DMA、缓存、时间戳和数据完整性检查？
	- **触发**：用 IMU 的数据就绪中断（DRDY）触发读取，不轮询。
	- **DMA**：SPI 收发都走 DMA，CPU 不搬数据；DMA 完成中断里只做通知。
	- **缓存**：双缓冲/环形缓冲——DMA 写一块，任务处理另一块，避免边写边读。
	- **时间戳**：在 DRDY 中断里用硬件定时器打戳（不在任务里打，避免调度抖动）。
	- **完整性**：帧计数器查丢帧、定期读 WHO_AM_I 确认芯片在线、数值量程合理性检查、丢帧/错误计数统计上报。

## 3.3 UART、CAN 与协议设计

1. 项目中的串口是怎么使用的？
	- 接收：**DMA + 空闲中断**收不定长帧，数据进环形缓冲，通知解析任务处理。
	- 发送：DMA 异步发送 + 发送完成中断，上层通过队列投递。
	- 用途：日志输出、与上位机的协议通信、外挂模块 AT 指令（按自己项目说）。
2. 串口协议栈怎么设计？怎么量化性能指标？
	- 分三层：**驱动层**（DMA/中断收发、环形缓冲）→ **帧层**（帧头+长度+CRC，状态机解析、错位重同步）→ **应用层**（命令分发、响应）。
	- 指标：吞吐率（实际字节/s 对比波特率理论值）、丢帧率、误码率、单帧解析耗时、缓冲区峰值占用、命令响应延迟。
	- 验证：高速连发压测 + 错误注入（截断帧、错 CRC）看能否正确丢弃并恢复同步。
3. CAN 总线及其优势。
	- 多主架构、差分双绞线，最高 1 Mbps（CAN FD 更高），基于 ID 的**非破坏性仲裁**（ID 越小优先级越高）。
	- 优势：差分信号**抗干扰强**；硬件 CRC + ACK + 自动重发；**错误隔离**（错误计数器，故障节点自动 bus-off）；多主无需轮询；两根线布线简单。适合汽车、工业等强干扰场景。
4. UART/CAN/USB 通信出现偶发丢包，你会如何区分是物理层问题、驱动问题、协议设计问题还是上层处理不及时？
	- **加序号定位**：发端加递增序号，收端统计断号发生的位置和规律——这是分层排查的抓手。
	- **物理层**：示波器看波形质量、幅值、干扰；查地线、终端电阻、波特率偏差（>2% 就危险）。
	- **驱动层**：查 ORE 溢出标志、错误中断计数，DMA 是否来不及搬运。
	- **协议层**：有无校验和重同步机制，帧边界错位后能否恢复。
	- **上层**：接收缓冲/队列是否溢出；临时加大缓冲或提高任务优先级做对照实验，丢包消失即上层问题。
5. 让你设计一个上位机和嵌入式设备之间的二进制协议，你会如何考虑帧格式、校验、序号、重传、兼容性和调试便利性？
	- **帧格式**：帧头(2 字节魔数) + 版本 + 命令字 + 序号 + 长度 + 数据 + CRC16。
	- **校验**：CRC16 覆盖长度和数据，防错位漏检。
	- **序号**：检测丢包、去重、请求-响应匹配。
	- **重传**：ACK + 超时重传，限制最大次数后报错。
	- **兼容性**：版本字段 + TLV 可扩展字段，旧设备忽略不认识的字段。
	- **调试**：统一错误码、心跳帧、写个抓包解析脚本让协议肉眼可读。
6. 总线协议了解哪些？I²C 和 SPI。
	- **I²C**：2 根线、半双工、靠地址寻址可挂多设备、有 ACK 应答、速率低（100k/400k）、协议开销大。
	- **SPI**：4 根线、全双工、靠 CS 选设备、无应答机制、速率高（可达几十 Mbps）、协议简单。
	- 选型：引脚紧张、低速多设备用 I²C；高速大数据量（Flash、屏、IMU）用 SPI。
	- 另外了解：UART（异步、点对点）、CAN（多主差分、车载）、USB、RS485。

## 3.4 GPIO、定时器与传感器

1. 定时器用过吗？常见工作模式有哪些？
	- **基本定时**：更新中断做时基/周期任务。
	- **PWM 输出**（输出比较）：电机调速、LED 调光、舵机。
	- **输入捕获**：测频率、脉宽（如超声波回波）。
	- **编码器接口模式**：电机测速测向。
	- 其他：单脉冲模式、定时器级联、触发 ADC/DMA。
2. STM32 中 GPIO 引脚配置为输入模式时，上拉/下拉电阻的作用是什么？
	- 输入悬空时引脚是高阻态，电平不确定、易受干扰误触发；上/下拉给一个**确定的默认电平**。
	- 上拉默认高电平（典型：按键另一端接地）；下拉默认低电平。
	- 还能避免引脚停在中间电平造成额外功耗；I²C 开漏输出必须外接上拉。
3. 一个传感器驱动偶发读数异常，你会从硬件连接、总线时序、DMA、中断、数据滤波、任务调度哪些角度排查？
	- **先收集规律**：异常时保存原始数据和上下文，看是否与温度、时间、特定操作相关。
	- **硬件**：电源纹波、地线、接插件接触、走线长度和干扰、上拉阻值。
	- **总线时序**：示波器抓异常瞬间波形，时钟速率是否超标、时序参数是否满足手册。
	- **DMA**：缓冲被覆盖、读取时 DMA 还没传完、缓存一致性。
	- **中断**：被高优先级抢占导致读取超时、标志位清除错误。
	- **滤波**：限幅剔野值 + 均值/低通，区分真实噪声还是通信错误（通信错误应修根因，不能只靠滤波盖住）。
	- **调度**：读取任务被延迟导致传感器 FIFO 溢出，提高优先级或改中断驱动。

# 4 中断与 DMA

1. 中断如何产生？中断响应的基本过程是什么？
	- **产生**：外设事件（GPIO 边沿、定时器溢出、串口收到数据）→ 置位中断标志 → 若中断已使能，向 NVIC 发请求。
	- **响应**：NVIC 判优先级决定是否抢占 → 当前指令执行完 → 硬件自动压栈 8 个寄存器（xPSR、PC、LR、R12、R3-R0）→ 从向量表取 ISR 地址跳转执行 → ISR 内清标志 → 退出时硬件自动出栈恢复现场。
2. 外部中断的触发方式有哪些？中断优先级配置需遵循什么原则？
	- 触发方式：**上升沿、下降沿、双边沿**（STM32 EXTI）；部分平台还有电平触发。
	- 原则：越紧急越高（安全保护 > 实时控制 > 通信 > 普通）；STM32 **数值越小优先级越高**。
	- 需要调用 RTOS API 的中断，优先级不能高于 `configMAX_SYSCALL_INTERRUPT_PRIORITY`。
	- 高优先级 ISR 必须短；同类中断用子优先级区分，避免无意义的嵌套。
3. 中断管理如何设计？
	- ISR 只做最小工作：清标志、取数据、发通知（信号量/任务通知），耗时处理交给任务——"上半部/下半部"思想。
	- 全局优先级规划表，统一分配、文档化，避免各模块随意配。
	- 临界区关中断时间尽量短；与 ISR 共享的变量用 `volatile` + 原子访问。
4. 中断服务函数与任务间通信的安全方式是什么？为何不建议在 ISR 中直接调用复杂函数？
	- 安全方式：**FromISR 版本 API**——任务通知（最快）、`xSemaphoreGiveFromISR`、`xQueueSendFromISR`，配合 `portYIELD_FROM_ISR` 请求切换。
	- 不建议复杂函数的原因：ISR 执行期间阻塞同级/低级中断和任务调度，破坏实时性；**ISR 中不允许阻塞**，很多 API 会阻塞；printf/malloc 等函数不可重入；ISR 栈（MSP）使用不可控，易溢出。
5. 中断里面可以用 FreeRTOS 函数吗？（一般优先级数值小于 5 就不可以）
	- 可以，但只能用 **FromISR 后缀**的 API。
	- 且该中断的优先级数值必须 **≥ configMAX_SYSCALL_INTERRUPT_PRIORITY（通常配为 5）**；数值小于 5（逻辑优先级更高）的中断不允许调用任何 FreeRTOS API。
6. 为什么优先级数值小于 5 不可以？（与 `BASEPRI` 和 `configMAX_SYSCALL_INTERRUPT_PRIORITY` 有关）
	- FreeRTOS 进临界区不是全关中断，而是写 **BASEPRI = 5**，只屏蔽优先级数值 ≥ 5 的中断。
	- 数值 < 5 的中断**不受屏蔽**，可能在内核正在操作链表/队列时打断它；此时若调用 FreeRTOS API，会并发修改内核数据结构，导致数据损坏、系统崩溃。
	- 这样设计的好处：最高优先级中断永不被 OS 延迟（超低延迟），代价是不能用 OS API。
7. DMA 全满中断和串口空闲中断，当二者都满足触发条件时，会同时将标志位置位然后触发吗？
	- 会各自置位：DMA 传输完成标志（TC）在 DMA 控制器里，IDLE 标志在 UART 状态寄存器里，是**两个独立的中断源**。
	- 两个中断按 NVIC 优先级**先后依次执行**（不会真正"同时"），标志位都不会丢。
8. 满足 DMA 和空闲中断的触发条件后，如果把中断关闭 10 个字节的时间再打开，或者被更高优先级中断阻塞，两个标志位都还在，这时该怎么处理？
	- 标志会以挂起态（pending）保留，开中断后按优先级依次进 ISR，不会丢。
	- 正确做法：**不依赖"哪个中断先来"**，数据边界统一以 DMA 写位置为准——在 ISR 里读 NDTR 计算当前写指针，与上次处理位置之差就是新数据长度。
	- 用环形缓冲管理，中断只是"触发点"，长度由指针计算得出；这样即使中断延迟、顺序颠倒或两个都 pending，处理逻辑依然正确（第二个中断进来时算出长度为 0，直接返回即可）。

# 5 RTOS 与操作系统

## 5.1 为什么使用 RTOS

1. 裸机系统与 RTOS 系统的核心差异是什么？有用 RTOS，为什么要用？（可移植性、资源受限下完成并发任务、可扩展性等角度）
	- **裸机**：大循环 + 中断，所有功能耦合在一起，实时性靠人工安排执行顺序，功能一多就难维护。
	- **RTOS**：抢占式调度，高优先级任务及时响应；任务独立解耦；提供队列/信号量等通信机制；延时阻塞不空耗 CPU。
	- 为什么用：并发任务多（通信 + 控制 + 显示同时跑）；实时性有硬指标；模块化便于团队协作和移植；加功能只需加任务，可扩展性好。
	
2. RTOS 是如何保证实时性的？（任务优先级、中断、锁、信号量、任务调度等）
	- **抢占式优先级调度**：高优先级任务一旦就绪立即抢占 CPU。
	- 内核关中断时间极短、中断延迟低且可确定。
	- 互斥锁带**优先级继承**，防优先级反转。
	- 信号量/任务通知让任务事件驱动，事件到达立刻唤醒。
	- SysTick 提供确定的时基；同优先级时间片轮转保证公平。

## 5.2 任务状态与调度

1. RTOS 中任务的状态有哪些？
	- **运行 Running**：正在占用 CPU（单核同时只有一个）。
	- **就绪 Ready**：随时可运行，等调度器选中。
	- **阻塞 Blocked**：等延时、信号量、队列等事件，超时或事件到达后回就绪。
	- **挂起 Suspended**：被 `vTaskSuspend` 挂起，必须显式 Resume 才能回来。
2. RTOS 任务运行机制是什么？RTOS 如何保证任务调度？（任务调度器、优先级、时间片轮转、上下文切换、阻塞唤醒等）
	- 每个任务有独立的**栈**和 **TCB**（任务控制块，存 SP、优先级、状态）。
	- 调度器永远从就绪表中选**最高优先级**任务运行；同优先级按时间片轮转。
	- SysTick 周期中断驱动时基（延时到期、时间片切换）；实际切换由 **PendSV** 异常完成。
	- 任务等资源时主动进阻塞态让出 CPU，事件到达（含 ISR 中 give）被唤醒回就绪表，触发一次调度。
3. 高优先级任务是如何抢占低优先级任务的？
	- 某事件（中断、API 调用、延时到期）使高优先级任务变就绪 → 内核在调度点发现"就绪的最高优先级 ≠ 当前任务" → 悬起 PendSV。
	- PendSV 里：保存当前任务上下文到它自己的栈 → SP 存入 TCB → 换成高优先级任务的 SP → 恢复其上下文 → 运行。
	- 常见抢占点：ISR 退出时、API 调用内、SysTick。
4. 任务优先级的分配原则是什么？高优先级任务长期占用 CPU 会导致什么问题？
	- 原则：按紧急程度/截止期分配；**周期越短优先级越高**（Rate Monotonic）；高优先级任务必须"短平快"且有阻塞点；CPU 密集型放低优先级。
	- 长期占用的后果：低优先级任务**饿死**、通信超时、看门狗复位、系统假死。
	- 规避：高优先级任务必须事件驱动（阻塞等待），禁止死循环轮询。
5. 如果你负责一个飞控或云台控制相关模块，如何保证 1 ms / 5 ms / 10 ms 级周期任务的实时性？
	- 优先级按频率分：1 ms 最高、5 ms 次之、10 ms 再次（RMS 原则）。
	- 用 `vTaskDelayUntil`（绝对延时）保证周期不累积漂移，或硬件定时器触发。
	- 中断里只做采集 + 打时间戳，运算放任务里做。
	- 关键路径避免锁竞争：共享数据用**双缓冲/原子交换**，临界区极短；控制回路内不做动态内存分配、不打同步日志。
	- 验证：GPIO 翻转 + 示波器统计周期抖动（jitter），监控 CPU 占用率和栈水位。
6. Ucos 滴答时钟设置频率。
	- 由 `OS_TICKS_PER_SEC` 配置，一般 **100~1000 Hz**（tick = 10 ms ~ 1 ms），常用 1000 Hz。
	- 权衡：太高 → tick 中断开销大；太低 → 延时分辨率粗、响应慢。

## 5.3 任务通信与同步

1. RTOS/FreeRTOS 任务之间通信是怎么做的？用过哪些方式、什么场景？（任务通知、消息队列、信号量、互斥锁；平时用什么比较多？）
	- **任务通知**：最快、最省 RAM，一对一同步/轻量计数，替代二值信号量。
	- **消息队列**：带数据的通信，如串口 ISR → 解析任务。
	- **信号量**：二值做同步/中断唤醒，计数做资源计数。
	- **互斥锁**：保护共享资源，带优先级继承。
	- **事件组**：等多个事件的组合（与/或）。
	- 平时最多：队列传数据 + 任务通知做同步（结合自己项目举例）。
2. 消息队列的发送与接收过程是什么？如何处理队列满/空的异常情况？
	- 发送：数据**拷贝**进队列尾部（大数据可只发指针）；接收：从队头拷出；两端都可设阻塞超时。
	- 队列满：立即返回失败 / 阻塞超时 / 永久阻塞，三选一；ISR 中发送不允许阻塞，失败要**记丢包计数**。
	- 满的对策：状态类数据用 `xQueueOverwrite` 覆盖旧值；否则加大队列或提高消费者优先级；失败计数上报便于发现设计问题。
	- 队列空：接收方阻塞等待即可，设超时防止永久卡死。
3. 信号量的两种类型分别适用于什么场景？
	- **二值信号量**：任务间同步、中断唤醒任务（ISR give，任务 take），只有 0/1 两态。
	- **计数信号量**：管理 N 个相同资源（资源池）；或累计事件次数（中断连发多次不丢）。
4. 信号量和互斥锁有什么区别？底层是怎么设计的？（信号量一般用于任务同步，互斥锁用于临界资源保护，互斥锁有优先级继承）
	- **用途**：信号量做同步（A 通知 B）；互斥锁做互斥（保护共享资源）。
	- **所有权**：互斥锁有持有者，谁上锁谁解锁；信号量无所有权，可以 A give、B take。
	- **优先级继承**：互斥锁有（防优先级反转），信号量没有。
	- **底层**：都是"计数 + 等待链表"——拿不到时任务挂到等待链表进入阻塞，释放时唤醒其中最高优先级的任务；互斥锁额外记录 holder，必要时把 holder 的优先级临时抬到等待者的水平。
5. 多任务/多线程运行时需要注意什么？线程同步用过什么？资源竞争如何通过同步和互斥保证安全？
	- 注意：共享数据竞争、死锁、优先级反转、函数可重入性（strtok 等不可重入）、每个任务栈大小单独评估。
	- 用过：互斥锁、信号量、事件组、任务通知（Linux 下：mutex、条件变量、读写锁）。
	- 保证安全：共享资源统一加锁访问，锁粒度小、持锁时间短；加锁顺序全局一致防死锁；与 ISR 共享的数据用关中断或原子操作。
6. 临界区数据处理：互斥锁与条件判断如何选择？
	- 临界区**极短**（几条指令，改个标志/计数）：直接关中断 `taskENTER_CRITICAL()`，开销最小。
	- 临界区**较长**或中途可能阻塞：用互斥锁，不影响中断响应。
	- 与 **ISR 共享**数据：只能关中断（或关对应优先级），不能用锁。
	- 单写单读的简单标志：`volatile` + 原子读写即可，不必加锁。
7. 一个控制任务和通信任务同时访问同一份状态数据，如何设计数据同步？什么时候用互斥锁，什么时候用消息队列或双缓冲？
	- **互斥锁**：结构体小、访问频率低，简单直接，注意持锁时间要短。
	- **双缓冲/乒乓 + 原子指针交换**：控制回路实时性苛刻，读写互不阻塞，最适合高频状态共享。
	- **消息队列**：事件/命令流式数据，天然解耦、自带缓冲。
	- 原则：**控制任务绝不能被通信任务长时间阻塞**——单向数据流用队列，大块共享状态用双缓冲，少量字段用锁。
8. 自旋锁和互斥锁有什么区别？
	- **自旋锁**：拿不到就原地忙等（烧 CPU），不睡眠、无上下文切换；用于**多核**、极短临界区、不可睡眠的上下文（中断）。
	- **互斥锁**：拿不到就睡眠让出 CPU，有切换开销；用于较长临界区。
	- 单核 RTOS 上自旋没有意义（自旋时持锁者根本没机会运行），等价做法是关调度/关中断。
	- 自旋锁临界区内禁止睡眠。
9. RTOS 的内存池机制如何优化内存分配？
	- 预先划好 N 个**固定大小的内存块**组成池，分配/释放就是取/还链表头，**O(1) 且时间确定**。
	- 无碎片问题，适合实时系统频繁分配释放。
	- 缺点：块大小固定有内部浪费，需按业务分几种规格的池。
	- 例：uC/OS `OSMemCreate`；FreeRTOS heap_4 的合并算法也是在缓解碎片。

## 5.4 优先级反转与上下文切换

1. 什么是优先级反转？FreeRTOS 中高优先级任务被低优先级任务间接阻塞，可能是什么问题？如何定位和规避？
	- 定义：低优先级任务持锁 → 高优先级任务等这把锁 → **中优先级任务抢占了低优先级任务**运行 → 高优先级被中优先级间接卡住，优先级"反转"。经典案例：火星探路者。
	- 定位：trace 工具（SystemView/Tracealyzer）看任务时序；发现高优先级长时间 Blocked 时查它等的锁被谁持有。
	- 规避:用**互斥锁的优先级继承**（持锁者临时升到等待者优先级）、优先级天花板、缩短临界区、减少高低优先级任务共享锁。
2. 了解任务上下文切换的具体过程吗？上下文切换时系统做了什么？线程切换是怎么设计的？
	- 触发 **PendSV**（配成最低优先级异常，保证不打断其他中断）。
	- 进异常时硬件自动压栈：xPSR、PC、LR、R12、R3-R0 → 当前任务栈。
	- 软件（汇编）再压 R4-R11，把 SP 存进当前任务 TCB。
	- 选出最高优先级就绪任务，从它的 TCB 取出 SP。
	- 出栈 R4-R11，异常返回时硬件自动恢复其余寄存器 → 新任务从上次断点继续跑。
3. 上下文切换源码中有汇编代码，了解哪些相关汇编指令？
	- `MRS`/`MSR`：读/写特殊寄存器（PSP、MSP、BASEPRI）。
	- `STMDB`/`LDMIA`：多寄存器批量压栈/出栈，如 `STMDB R0!, {R4-R11}`。
	- `CPSID`/`CPSIE`：关/开中断。
	- `BX LR`：异常返回（LR 中是 EXC_RETURN 特殊值）。
	- `ISB`/`DSB`：指令/数据同步屏障；`SVC`：系统调用，用于启动第一个任务。

## 5.5 进程、线程与通用调度

1. 进程和线程的区别。
	- 进程是**资源分配**的基本单位，有独立地址空间；线程是 **CPU 调度**的基本单位。
	- 同进程的线程共享地址空间（代码、堆、全局变量），各自有栈和寄存器。
	- 切换开销：进程大（切页表、TLB 失效）；线程小。
	- 隔离性：进程一个崩不影响其他；线程一个崩整个进程挂。
	- 通信：进程要 IPC；线程直接共享内存（但要同步）。
2. 进程间通信方式。
	- 管道（匿名管道/命名管道 FIFO）。
	- 消息队列。
	- **共享内存**（最快，需配合信号量做同步）。
	- 信号量、信号（signal）。
	- Socket（可跨机器）。
3. 多级反馈队列调度和时间片轮转。
	- **时间片轮转 RR**：就绪队列排队，每个进程跑一个时间片，用完排队尾；公平但不区分紧急程度。
	- **多级反馈队列 MLFQ**：多个优先级队列，优先级越高时间片越短；新进程进最高队列，用完整时间片就降级（CPU 密集型沉底），IO 密集/交互型留在高层；定期把所有进程提回最高层防饿死。
	- 效果：兼顾交互响应和整体吞吐。
4. `nice` 命令如何影响调度？
	- 调整进程的静态优先级，范围 **-20（最高）~ 19（最低）**，默认 0；负值需 root。
	- nice 值越高越"谦让"，分到的 CPU 时间越少；在 CFS 中体现为权重不同（vruntime 增长速度不同）。
	- 用法：`nice -n 10 cmd` 启动时指定；`renice` 修改运行中的进程。

# 6 固件升级、参数存储与可靠性

1. 固件/OTA 升级过程中突然断电，再次上电后如何保证系统能够恢复并重新启动升级流程？如何设计 Bootloader、固件校验、A/B 分区和回滚策略，保证断电不变砖？
	- **A/B 双分区**：新固件写到备区，写完并 CRC/签名校验通过后，才切换启动标志——升级中断电时标志未切换，仍从旧分区正常启动，重新升级即可。
	- **Bootloader 每次启动**：校验目标分区固件的 CRC/签名，无效就自动回滚到另一分区；BL 本身永不升级（或双份）。
	- **标志区原子化**：启动标志双份存储 + CRC + 递增序号，先写数据最后写有效标记，任何时刻断电都有一份可用。
	- **试运行机制**：新固件首次启动置 trial 标志，自检通过才 commit；不通过则看门狗复位后 BL 回滚。
	- 资源紧张的变体：BL + 单 APP 区 + 升级包暂存区（外部 Flash），BL 负责校验和搬运。
2. 如何设计一个可靠的参数存储模块，保证掉电、Flash 擦写失败、版本升级后参数仍然可以恢复？
	- **双备份区 + CRC**：先写主区、校验，再写备份区；读取时主区 CRC 错就用备份恢复。
	- **写入原子化**：新数据全部写好并校验后，最后才更新"有效标志/序号"，写一半断电不影响旧数据。
	- **磨损均衡**：日志式追加写（每条记录 = 键值 + 序号 + CRC），页满后垃圾回收换页；读取取"序号最新且 CRC 正确"的记录。
	- **版本兼容**：参数区带版本号，升级后新增字段用默认值补齐，不认识的字段跳过。
	- **兜底**：两区都损坏时加载出厂默认值并报警；擦写后回读校验，失败则换块重试 + 坏块标记。
