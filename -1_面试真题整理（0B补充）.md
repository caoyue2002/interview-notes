## 3.1 你在项目中如何实现六路RS485、29电机、1kHz的高频数据接收？

**回答：**

我们采用的是六路RS485总线，每路挂最多6个电机，总共29个，以1kHz刷新率收发数据。接收方案用的是DMA加IDLE中断，六路总线对应14个DMA通道，每路独立RX和TX各一个，DMA工作在单次非循环模式。总线上字节传输完、进入空闲状态时，USART的IDLE中断自动触发，ISR里再去解析缓冲区里的数据。

最开始遇到了一个丢帧问题。早期实现是IDLE中断触发之后，先停止DMA，解析完再重启DMA。这在低频下没问题，但1kHz高频下出bug了：电机采用了菊花链回复，回复帧是串行发的，motor_0回完大约50µs后motor_1紧接着发，而停DMA去解析的窗口期恰好也是这个量级，motor_1的字节在DMA关闭期间到达，USART硬件把它丢掉了，那一帧CRC校验失败，整帧被丢弃，上层任务超时。

解决方案是IDLE ISR里不再停止DMA，而是直接读DMA的DTCNT寄存器（剩余传输计数），用缓冲区总长度减去它得到当前写入位置，然后只解析上次处理位置到当前位置这个范围的数据，DMA继续往后写，ISR处理已有数据，两者互不干扰。还有一个细节，IDLE触发的瞬间USART的RDR寄存器可能还有一个字节没被DMA搬走，需要手动读出来追加到缓冲区末尾，否则帧尾字节会丢。解析时用增量扫描加不完整帧保留的策略，从SOF帧头开始读长度字段，如果当前范围内字节不够一帧就停在这里，下次IDLE触发时从这个位置继续，帧跨两次IDLE也不会出错。

RS485是半双工，发送前RE引脚拉高切TX模式，关掉IDLE中断，停DMA RX，启动DMA TX发送。发送完成的时机用的是TDC中断，这是硬件告诉你最后一个字节已经从移位寄存器发出去了的信号，在TDC ISR里立刻把RE拉低切回RX，重启DMA RX，释放一个信号量通知任务下次可以发了。这比软件延时精确得多，切换延迟在1µs以内。波特率选的6Mbaud，88字节的控制帧只需约147µs传完，1ms周期内收发都有充足余量。

可靠性上还有几个细节：重启DMA RX时必须通过先读STS再读DT的序列主动清除ORE（Overrun Error）标志，否则USART接收路径会被阻塞，后续字节全收不到；等待TDC信号量设了5ms超时，防止TDC中断丢失时任务死锁；读帧长度字段时用memcpy而不是直接强转指针，避免Cortex-M非对齐访问触发HardFault。六路同时运行，1kHz稳定跑下来没有超时和丢帧。

## 3.2通过SPI读取陀螺仪数据的策略是什么？

我们采用的ICM42670陀螺仪配置了400Hz ODR，配置INT引脚为DRDY（数据就绪），SPI频率为3.75Mhz，CPOL = 1,CPHA = 1,连接单片机PA8，触发EXTI外部中断上升沿，在中断里释放一个二值信号量，唤醒IMU任务进行采样。

采样的时候，SPI使用CPU轮询传输，因为ICM42670不支持SPI burst读，每个寄存器需要独立CS事务，DMA的优势不明显，反而增加复杂度，所以直接用轮询读完12字节的加速度和陀螺仪原始数据。任务阻塞在信号量上等DRDY，DRDY到来唤醒后立刻读取，延迟在5到10微秒以内。
时间戳方面，使用TMR1做微秒级计数，16位计数器配合溢出中断累积成64位时间戳，在DRDY中断触发的瞬间打上时间戳，和数据绑定传给上层，保证姿态融合算法拿到的dt是真实的采样间隔，而不是任务调度的延迟。

数据完整性方面，读取前检查INT_STATUS_DRDY寄存器的DRDY标志位，确认有新数据再读，避免在ODR更新窗口期间读到撕裂的数据。另外每次读完数据寄存器，DRDY标志会自动清除，如果下次信号量来了但DRDY没置位，说明发生了漏中断，直接跳过这次采样。如果想进一步保证，可以引入双缓冲，DMA写A缓冲区时上层从B区读，DMA写完后翻转指针切换读写角色，避免读到写了一半的数据。不过因为ICM42670不支持DMA burst，这个场景下我们直接用轮询读完再整块交给上层，本质上是原子操作，不存在撕裂问题。

## 3.2 项目中用的自定义V1帧是怎么设计的？

**V1普通包帧格式（13字节包头 + N字节数据 + 2字节CRC16，最大1023字节）：**

| 字节偏移 | 字段 | 位宽 | 值/说明 |
|---------|------|------|---------|
| 0 | SOF | 8bit | 固定 `0x55`（01010101，高低位交错，利于时钟同步和误码检测） |
| 1~2 高3位 | Ver | 3bit | 协议版本号，V1 = 1 |
| 1~2 中3位 | Attr | 3bit | bit0 = headerType（0=普通包/1=广播包），bit1-2保留 |
| 1~2 低10位 | Len | 10bit | 整帧总长度（含包头+数据+CRC），最大1023 |
| 3 | HeaderCRC8 | 8bit | 仅校验前3字节（SOF+Ver+Attr+Len），用于快速过滤包头损坏的帧 |
| 4 | AttrExt | 8bit | bit0 = cmdType（0=请求/1=应答），bit1 = needAck，bit2 = encType（预留） |
| 5 | SenderIndex | 8bit | 发送者模块序号（本项目MCU index=0x00） |
| 6 | SenderType | 8bit | 发送者模块类型（MCU_APP=0x06，HOST=0x00，ESC=0x02） |
| 7 | ReceiverIndex | 8bit | 接收者模块序号 |
| 8 | ReceiverType | 8bit | 接收者模块类型 |
| 9~10 | Seq | 16bit | 帧序列号，每帧递增，用于丢包检测和请求/应答绑定 |
| 11 | CmdSet | 8bit | 命令集（0x07=通用，0x09=人形机器人专用） |
| 12 | CmdId | 8bit | 命令ID（具体业务命令编号） |
| 13~13+N-1 | DATA | N字节 | 用户数据段，格式由CmdId定义，可为空 |
| 13+N~13+N+1 | CRC16 | 16bit | 整包校验，覆盖SOF到DATA末尾 |

**广播包（8字节包头，无路由字段）：**

| 字节偏移 | 字段 | 位宽 | 说明 |
|---------|------|------|------|
| 0 | SOF | 8bit | `0x55` |
| 1~2 | Ver+Attr+Len | 16bit | 同普通包，headerType=1 |
| 3 | HeaderCRC8 | 8bit | 同普通包 |
| 4~5 | Seq | 16bit | 序列号 |
| 6 | CmdSet | 8bit | 命令集 |
| 7 | CmdId | 8bit | 命令ID |
| 8~ | DATA + CRC16 | — | 同普通包 |

**双重校验设计：** HeaderCRC8只校验前3字节，接收端收到4字节就能判断包头合法性，无效帧立即丢弃不等完整帧；整包CRC16兜底保证数据完整性。两层校验分别针对"快速过滤"和"完整性保证"两个目标。

## 3.3 项目中如何保证与上位机通信的实时性？

**回答：**

我们上位机通信用的是UART7，波特率配的4Mbaud，236字节的feedback帧传输时间约0.59ms，1ms周期内单次收发都有余量。

任务优先级上，host_comm_task配的是5，仅次于电机控制任务的7，高于IMU任务的3和调试任务的1，保证上位机命令能及时被处理，不被低优先级任务饿死。

接收方面，UART7 RX用DMA1_CH1循环模式，开了一个4096字节的ring buffer，DMA全程持续写入不停止。任务主循环每次通过读DTCNT计算head位置，和tail做差得到新数据量，直接memcpy到线性缓冲区再解析帧。4096字节按500Hz来料可以缓冲约76ms，任务最坏调度延迟2ms，余量充足不会溢出丢帧。这里有意没用IDLE中断通知，因为任务本身2ms周期轮询已经足够，中断通知的收益不大，反而增加控制流复杂度。

发送方面，TX用了ping-pong双缓冲，两个缓冲区s_tx_buf[0]和s_tx_buf[1]，DMA在发buf[0]的时候，任务可以往buf[1]填下一帧数据，填完交换索引，等DMA完成信号量释放后立刻启动新传输。这样CPU填帧和DMA搬运完全并行，不会因为等DMA完成而空等。

命令应答不在协议解析回调里直接触发DMA发送，而是先把应答帧memcpy到一个pending缓冲区，置一个标志，等主循环的安全点process_rx返回之后再统一发送，避免在帧解析中途嵌套触发DMA配置，消除重入和死锁的风险。

feedback帧里还带了TMR1的64位微秒时间戳，TMR1是16位1µs计数器配合溢出中断累积成64位，读取时用双读法，不用关中断也能安全读到一致的值，时间戳精确反映数据采集时刻而不是任务调度时刻，上位机可以用它做精确的数据对齐。

### ESC电机固件升级：透传网关架构

```

  HOST(上位机) ─── UART7 4Mbaud ───► MCU(透传网关) ─── RS485 6Mbaud ───► ESC×29

MCU不是升级目标，是中间人:
  · 解析HOST的V1帧，识别路由目标
  · 修改帧头(清sender_index，重算CRC16)后转发到RS485

  │ 0x80 ~ 0xBF   │ 广播: 低6位=bus_mask (哪几路同时发)       │
  │ 0xFF          │ 广播全部6路 (alias for mask=0x3F)         │
  └────────────────┴──────────────────────────────────────────┘

```

## 5.1 任务切换时涉及的寄存器及其作用

Cortex-M 有 `R0` 到 `R15`，另有 `xPSR` 状态寄存器。任务被切走后，要让它能像从未被打断一样继续执行，就必须恢复该任务当时依赖的寄存器状态。

| 寄存器          | 主要作用                      | PendSV 切换中的处理                                                             |
| ------------ | ------------------------- | ------------------------------------------------------------------------- |
| `R0-R3`      | 临时数据、函数前 4 个参数、返回值        | 异常进入时由硬件自动压入任务 PSP 栈；异常返回时自动恢复。                                           |
| `R4-R11`     | 通用寄存器；按 AAPCS 属于被调用者保存寄存器 | PendSV 用汇编手动压入/恢复。                                                        |
| `R12`        | 临时寄存器（IP）                 | 异常进入时硬件自动压入，异常返回时自动恢复。                                                    |
| `R13` / `SP` | 栈指针                       | 任务在线程模式通常用 `PSP`；最终的 `PSP` 保存到 `TCB->pxTopOfStack`。异常 Handler 自己使用 `MSP`。 |
| `R14` / `LR` | 正常函数调用时保存返回地址             | 异常处理时保存 `EXC_RETURN`，它告诉硬件异常返回时使用哪种栈和栈帧。任务原来的 `LR` 位于硬件自动栈帧中。             |
| `R15` / `PC` | 下一条要执行的指令地址               | 异常进入时自动压栈；异常返回时恢复。恢复后的 PC 决定任务从哪里继续执行。                                    |
| `xPSR`       | 条件标志、Thumb 状态和当前异常信息      | 异常进入时自动压栈；异常返回时恢复。                                                        |

这里的“现场”不是逐个保存局部变量。局部变量原本就在该任务自己的栈中；若某个局部值暂存在寄存器中，它会随着寄存器现场一起被保留。

这其中，R0, R1, R2, R3, R12, LR, PC, xPSR由硬件自动保存，R3-R11由软件保存
