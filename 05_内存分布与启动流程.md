# 2 编译产物

程序编译后分为 4 部分，每部分存放位置不同：

| Keil 术语     | 标准段名      | 内容                                                    | 存放位置                       |
| ----------- | --------- | ----------------------------------------------------- | -------------------------- |
| **CODE**    | `.text`   | 函数体的机器指令                                              | Flash                      |
| **RO-DATA** | `.rodata` | read only数据: const 变量、字符串常量等只读数据                      | Flash                      |
| **RW-DATA** | `.data`   | read write数据: 已初始化且**非0**的全局/静态变量                     | **Flash（存初值）+ SRAM（运行副本）** |
| **ZI-DATA** | `.bss`    | zero init数据:未初始化或初始化为0的全局/静态变量，**+ 堆(heap)和栈(stack)** | 仅 SRAM                     |

> 没被调用的函数依然会被编译进 `.text`。

# 3 内存重映射与向量表

CPU 上电后**永远从 `0x00000000` 开始取指**，定为MSP地址。这个地址实际指向哪里，由 **BOOT0/BOOT1** 引脚决定：

| BOOT0 | BOOT1 | 启动位置                            | 物理地址         |
| ----- | ----- | ------------------------------- | ------------ |
| 0     | x     | **Flash**                       | `0x08000000` |
| 1     | 0     | 片内系统存储器（内置Bootloader，串口/USB下载用） | `0x1FFF0000` |
| 1     | 1     | **SRAM**                        | `0x20000000` |

> 最常用 **BOOT0=0 → Flash 启动**，重映射本质是硬件电路把 `0x00000000` 动态指向上面三者之一，由 BOOT 引脚电平在上电瞬间锁存决定。

---

## 3.1 中断向量表

无论重映射到哪，CPU 都从（重映射后的）`0x00000000` 开始，读取**中断向量表**最前面两个固定位置。
MSP 和 PC 是向量表**最前面两项**，完整的向量表是一整张表，**每4字节存一个中断/异常的入口地址**，紧接着排列在 Flash 起始处：

| 偏移       | 内容                                      |
| -------- | --------------------------------------- |
| `0x00`   | **MSP 初始值**（栈顶地址，不是中断入口，是唯一的特例）         |
| `0x04`   | **Reset_Handler** 地址                    |
| `0x08`   | NMI_Handler 地址                          |
| `0x0C`   | **HardFault_Handler** 地址                |
| `0x10`   | MemManage_Handler 地址                    |
| `0x14`   | BusFault_Handler 地址                     |
| `0x18`   | UsageFault_Handler 地址                   |
| ...      | ...（保留位）                                |
| `0x2C`   | SVC_Handler 地址                          |
| `0x38`   | PendSV_Handler 地址                       |
| `0x3C`   | SysTick_Handler 地址                      |
| `0x40` 起 | 各外设中断（USART、TIM、DMA、CAN...），具体顺序由芯片型号决定 |

# 5 SRAM 内存布局

## 5.1 裸机情况下

```
0x2000xxxx  (栈顶)
			┐
            │ 栈 Stack       （局部变量、返回地址、中断现场）
            │                （MSP/PSP 在这里活动）
            ├──────
            │ Free Space段   （堆和栈之间的缓冲区域）
            ├──────
            │ 堆 Heap        （动态内存分配区）
            ├──────
            │ .bss 段        （ZI_DATA）
            ├──────
            │ .data 段       （RW_DATA)
            ┘ 
0x20000000 (SRAM起始地址)
```

| 区域          | 存放内容                               |
| ----------- | ---------------------------------- |
| **栈 Stack** | 局部变量、函数调用的返回地址、压栈保存的寄存器等临时数据       |
| FreeSpace段  | 堆和栈之间预留的缓冲区，防止两者相撞                 |
| **堆 Heap**  | `malloc`动态分配的内存（系统堆，C库的堆）          |
| `.bss`      | 未初始化或初始化为0的变量（ZI_DATA），启动时清零       |
| `.data`     | 已初始化的全局/静态变量（RW_DATA），从Flash拷贝初值过来 |
## 5.2 FreeRTOS ：`.bss` 里的第二层堆

跑 FreeRTOS 后，FreeRTOS 自己管理一片堆空间，叫 **`ucHeap`**，本质是一个**未初始化的静态数组**：

> [!important] **`ucHeap` 这个变量本身是未初始化的静态数组，所以被编译器和其他普通的未初始化全局变量一起，归类进了 `.bss` 段**。
> ```
> .bss段（ZI_DATA区）
> 	├── 普通的未初始化/0初始化全局变量
> 	└── ucHeap这个静态数组（FreeRTOS的堆）
> ```

每动态创建一个任务，或者使用`pvPortMalloc()`函数，FreeRTOS 就在这片 `ucHeap` 里分配出一块内存，内存管理方式以heap算法为主。

```
ucHeap 内部布局（举例）：

0x20xx xxxx（高地址）
            ┐
            │ Task1 的 TCB
            ├──────
            │ Task1 的栈
            ├──────
            │ Task2 的 TCB
            ├──────
            │ Task2 的栈
            ├──────
            │ Task3 的 TCB
            ├──────
            │ Task3 的栈
            ├──────
            │ 用户分配的内存（动态数组）
            ├──────
            │ 未分配的内存
            ┘
0x20xx xxxx（低地址）
```

| 内容                              | 是什么                                             |
| ------------------------------- | ----------------------------------------------- |
| **任务的栈**                        | 该任务运行时的局部变量、函数调用、保存的寄存器现场——和裸机的栈作用一样，只是专属给这一个任务 |
| **任务的 TCB**（Task Control Block） | FreeRTOS用来描述这个任务的结构体：任务状态、优先级、栈指针的当前值、任务名等管理信息  |

> [!important] PSP 在这里活动 **PSP（进程栈指针）就在每个任务自己的栈里活动**——任务切换时，FreeRTOS 把 PSP 指向当前正在运行的任务的栈。MSP 不参与任务的运行（任务运行时CPU实际用的是PSP），**MSP 只在异常/中断处理时才被启用**。

## 5.3 动态创建 vs 静态创建

```c
xTaskCreate(...)                              // 动态：从ucHeap切
xTaskCreateStatic(..., xStack, &xTaskBuffer)  // 静态：用户自己的全局变量
```

|           | 动态 (xTaskCreate) | 静态 (xTaskCreateStatic) |
| --------- | ---------------- | ---------------------- |
| 内存来自      | 运行时从 `ucHeap` 切  | 编译期固定（用户声明的全局变量）       |
| 在ucHeap里吗 | ✅ 是              | ❌ 不是，直接进`.bss`，平级独立    |
| 会分配失败吗    | 可能（ucHeap耗尽）     | 不会                     |
| 优势        | 灵活，可运行时增删任务      | 内存确定，执行时间可预测           |

> [!warning] 不能说"RTOS堆栈都在ucHeap里" —— 只有动态创建的才是

# 6 Reset_Handler 执行的工作

CPU 跳转到 PC 指向的 `Reset_Handler` 后，开始执行真正的启动代码（这部分是软件，体现在 `startup_xxx.s` 里）：

```
Reset_Handler 依次做：
1. SystemInit()        → 配置系统时钟（HSE/HSI→PLL→SYSCLK）
2. 拷贝 .data           → 把RW-DATA初值从Flash搬到SRAM
3. 清零 .bss            → 把ZI-DATA所在SRAM区域写0
4. 跳转 main()
```

# 8 Q&A

## 8.1 RW-DATA 为什么"既在Flash又在SRAM"（高频考点）

```c
int speed = 100;   // 已初始化、非0 → RW-DATA
```

- **编译时**：初值 100 要被"记住"，存进 **Flash**（SRAM 掉电会丢，必须有个地方保存初值）
- **启动时**：`Reset_Handler` 把这个 100 从 Flash **拷贝**到 SRAM 对应地址
- **运行时**：CPU 读写的是 **SRAM** 里那份，可以被修改

> **RW-DATA 在 Flash 里占的是"初值的备份"，在 SRAM 里占的是"运行时真正使用的那份"——同一个变量，两边都占空间，但用途不同。**

---

## 8.2 ZI-DATA 为什么不占Flash

```c
int counter;          // 未初始化 → ZI-DATA
int sum = 0;           // 初始化为0 → ZI-DATA（和"未初始化"同等对待）
```

- 不需要记住任何初值（反正是0），**Flash 里不存任何内容**
- 启动时，`Reset_Handler` 只需要把这块 SRAM 区域**清零**即可
- **堆（malloc用的空间）和栈（函数调用、局部变量用的空间）也算在 ZI-DATA 里**，同样不占 Flash，只在 SRAM 划一块区域

## 8.3 为什么.data和.bss要分开，不能合并？

因为.data需要存初值（占用Flash），.bss不需要（只要清零）。分开是为了节省Flash空间——如果合并，所有变量都要在Flash存一份初值，会浪费大量Flash。
